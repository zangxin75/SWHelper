# SolidWorks 2026 API 自动化设计 Agent

## 项目概述
基于 Claude Code + Python + SolidWorks MCP Server + SW 2026 API 的对话式设计自动化系统。

## 技术栈
- Python 3.10+
- SolidWorks 2026 API (COM 接口)
- SolidWorks MCP Server (106 工具)
- Claude Code SDK / Anthropic API
- pytest + pytest-asyncio

## 项目路径
- 项目根目录: `D:\sw2026`
- 业务代码: `代码/Python脚本/`
- 测试代码: `代码/测试代码/`
- 需求文档: `文档/需求/`
- 配置文件: `配置/`
- MCP 服务器: `SolidworksMCP-python/`

## 开发流程规范（需求驱动开发 RDD）

**完整模板见**: `文档/项目管理/需求驱动开发模板_RDD.md`

### 强制规则

1. **任何开发任务必须按 "需求表 → 测试 → 业务代码" 的顺序执行**
2. **禁止在没有需求表的情况下编写业务代码**
3. **禁止在测试全红之前编写业务代码**
4. 需求表存放在 `文档/需求/` 目录，文件名格式 `req_{模块名}.md`
5. 测试代码存放在 `代码/测试代码/` 目录，文件名格式 `test_{模块名}.py`
6. 业务代码存放在 `代码/Python脚本/` 目录，不允许混入测试文件

### 执行步骤

```
步骤1: 解析项目方案 → 拆分为独立功能模块
步骤2: 为每个模块编写需求表（输入→预期输出的表格）
步骤3: 将需求表转化为 pytest 参数化测试
步骤4: 运行测试，确认全部为红色（失败）
步骤5: 编写/修改业务代码，逐个让测试变绿
步骤6: 所有测试绿色 = 需求实现完毕
```

### 需求表规范

- 每行必须包含: 编号、场景描述、具体输入值、具体预期输出值
- 输入必须是可复现的具体值，不能用模糊描述
- 预期输出必须是可断言的具体值，不能用"正确结果"等模糊表述
- 每个功能至少覆盖: 2个正常场景 + 1个边界场景 + 1个异常场景
- 编号规则: 模块缩写 + 序号，如 I-01, D-01, E-01

### 需求表格式

```markdown
| 编号 | 场景描述 | 输入 | 预期输出 | 备注 |
|------|---------|------|---------|------|
| XX-01 | 正常场景 | 具体输入值 | 具体预期输出值 | |
| XX-02 | 边界场景 | 具体输入值 | 具体预期输出值 | |
| XX-03 | 异常场景 | 具体输入值 | 具体预期输出值 | |
```

### 测试规范

- 使用 pytest + pytest-asyncio
- 使用 @pytest.mark.parametrize 将需求表转化为参数化测试
- 每个测试必须标注对应的需求编号（ids 参数）
- 外部依赖（SolidWorks API、Claude API）使用 mock
- 测试文件头部注释标注对应的需求文件路径

### 编码规范

- 一次只实现一个需求，按编号顺序
- 最简实现优先，不实现需求表以外的功能
- 编码顺序按依赖关系: 知识库 → 意图理解 → 任务分解 → 任务执行 → 结果验证 → 完整流程 → Claude集成
